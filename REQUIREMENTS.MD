# crux — Requirements

## What is crux?

crux is a personal project manager distributed as an MCP Bundle (.mcpb). The name comes from climbing: the **crux** is the hardest move on a route — the thing everything else depends on. In Critical Path Method (CPM), that is the critical path.

crux is a **hybrid tool**: it runs as a plain CLI in the terminal (free, instant, no AI), delegates simple reasoning to a local LLM (free, local), and exposes an MCP server for Claude to use when genuine intelligence is needed (paid, cloud, heavy lifting only). It ships as `crux.mcpb` for one-click installation and works in any git repo or standalone project.

---

## Design Goals

- Minimal 3rd-party dependencies to reduce attack surface
- Works without an LLM — CLI is fully self-sufficient
- Global project state, repo-scoped views — no cross-contamination
- Generated files (tasks.md, ADRs, reports) are outputs from the DB, never the source of truth
- One-click install via MCPB format; no npm publish, no certification required
- Browser UI via local HTTP server — no VSCode extension certification required

## Runtime & Language

| Decision | Choice | Reason |
|---|---|---|
| Runtime | Node.js v25 | Native TypeScript (no flags), native `node:sqlite`, native `node:test`, native `fetch()` |
| Language | TypeScript (native, no compile step) | Entry point is `.ts` directly; Node 25 strips types natively |
| Database | `node:sqlite` (stdlib) | No native bindings, no 3rd party, ships with Node 25 |
| HTTP | `node:http` (stdlib) | Browser UI + local API server |
| Testing | `node:test` (stdlib) | No Jest, no Vitest |
| 3rd party deps | `@modelcontextprotocol/sdk` only | MCP protocol — unavoidable |
| Dev environment | Apple `container` CLI + `node:25-slim` | Isolation from host npm/node issues |

## Data Architecture

### Global DB — source of truth

```
~/.crux/crux.db        ← single SQLite DB, owns ALL projects
~/.crux/config.json    ← user config (local LLM endpoint, GH token, rate, etc.)
```

### Per-repo pointer (optional, for code repos)

```
.crux/project.json     ← { "project_id": "uuid" } — pointer only, not data
```

When `crux` runs inside a repo it reads `.crux/project.json` and scopes all queries to that `project_id`. For non-repo projects (articles, research) there is no folder — just a DB record.

### Generated outputs (committed, human-readable, not source of truth)

| File | Generated by | Source |
|---|---|---|
| `tasks.md` | `crux report tasks` | `tasks` table |
| `docs/status-{date}.md` | `crux report status` | tasks + audit + CPM |
| `docs/adr/ADR-NNNN-*.md` | `crux report adrs` | `adrs` + `task_adrs` tables |
| `docs/milestone-phase-N.md` | auto on phase completion | `tasks` + milestone logic |

## Schema

```sql
CREATE TABLE projects (
    id           TEXT PRIMARY KEY,   -- UUID
    name         TEXT NOT NULL,
    type         TEXT NOT NULL,      -- code_repo | article | research | freelance | learning | personal
    status       TEXT NOT NULL DEFAULT 'active',  -- active | stalled | paused | done | dropped
    gh_repo      TEXT,               -- owner/repo or NULL
    gh_sync      INTEGER DEFAULT 0,  -- 0 = off, 1 = on
    sheets_id    TEXT,               -- Google Sheets ID or NULL
    hourly_rate  REAL,               -- for freelance projects
    created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE tasks (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id       TEXT NOT NULL REFERENCES projects(id),
    slug             TEXT NOT NULL,
    title            TEXT NOT NULL,
    description      TEXT,
    phase            TEXT,
    status           TEXT NOT NULL DEFAULT 'open',  -- open | in-progress | blocked | done | dropped
    priority         INTEGER DEFAULT 0,
    duration_days    REAL,
    early_start      REAL,
    early_finish     REAL,
    late_start       REAL,
    late_finish      REAL,
    float_days       REAL,
    is_critical      INTEGER DEFAULT 0,
    gh_issue_number  INTEGER,
    adr_ref          INTEGER REFERENCES adrs(id),
    coverage_target  REAL,
    created_at       TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(project_id, slug)
);

CREATE TABLE dependencies (
    predecessor_id INTEGER NOT NULL REFERENCES tasks(id),
    successor_id   INTEGER NOT NULL REFERENCES tasks(id),
    PRIMARY KEY (predecessor_id, successor_id)
);

CREATE TABLE sessions (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id  TEXT NOT NULL REFERENCES projects(id),
    started_at  TEXT NOT NULL,
    ended_at    TEXT,
    note        TEXT,
    minutes     REAL  -- computed on end
);

CREATE TABLE roi_records (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id  TEXT NOT NULL REFERENCES projects(id),
    recorded_at TEXT NOT NULL DEFAULT (datetime('now')),
    amount      REAL NOT NULL,
    currency    TEXT DEFAULT 'AUD',
    kind        TEXT NOT NULL,   -- revenue | cost | expected
    probability REAL DEFAULT 1.0,
    note        TEXT
);

CREATE TABLE test_runs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id  TEXT NOT NULL REFERENCES projects(id),
    task_slug   TEXT REFERENCES tasks(slug),
    run_at      TEXT NOT NULL DEFAULT (datetime('now')),
    phase       TEXT,            -- build | test-c | test-python | lint
    status      TEXT NOT NULL,  -- pass | fail
    coverage    REAL,
    output      TEXT,
    commit_sha  TEXT
);

CREATE TABLE audit (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id  TEXT REFERENCES projects(id),
    task_id     INTEGER REFERENCES tasks(id),
    event       TEXT NOT NULL,
    detail      TEXT,
    actor       TEXT,   -- human | crux-auto | claude
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE adrs (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id   TEXT NOT NULL REFERENCES projects(id),
    number       INTEGER NOT NULL,
    title        TEXT NOT NULL,
    status       TEXT NOT NULL DEFAULT 'proposed',  -- proposed | accepted | deprecated | superseded
    context      TEXT,
    decision     TEXT,
    consequences TEXT,
    created_at   TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(project_id, number)
);

CREATE TABLE task_adrs (
    task_id INTEGER REFERENCES tasks(id),
    adr_id  INTEGER REFERENCES adrs(id),
    PRIMARY KEY (task_id, adr_id)
);
```

## Dual-Mode Entry Point

`index.ts` detects its execution context:

```
process.stdin.isTTY === true  → parse argv, run as CLI (direct, free, no AI)
process.stdin.isTTY === false → start MCP stdio server (called by Claude or any MCP host)
```

Same underlying `lib/`. Same DB. Different interface.

## CLI Commands

All commands are free, instant, no LLM involved.

```
crux init [--from tasks.md]       Bootstrap DB for current repo project
crux status                        Current project: task counts, next unblocked, blockers
crux overview                      Meta Kanban: all projects, status, ROI, spread warning
crux task add <slug> <title>       Add task to current project
crux task done <slug> [--note]     Mark task done, auto-audit logged
crux task start <slug>             Mark task in-progress
crux task block <slug> [--note]    Mark task blocked
crux dep add <pred> <succ>         Add dependency edge
crux cpm                           Run critical path, print path + floats
crux graph [--dot]                 ASCII dependency DAG or DOT format
crux report tasks                  Regenerate tasks.md
crux report status                 Regenerate docs/status-{date}.md
crux report adrs                   Regenerate docs/adr/*.md
crux sync [--apply]                Dry-run or apply: DB ↔ GH Issues reconciliation
crux ready                         Go/no-go release readiness check
crux session start                 Start time tracking session for current project
crux session end [--note]          End session, log minutes
crux roi add <amount> [--kind]     Log revenue/cost against project
crux roi report                    ROI analysis + continue/pause/drop signal
crux spread                        Active project count + focus health warning
crux test-run <phase> <status>     Record build/test pass-fail + coverage
crux milestone check               Detect phase completions, auto-generate milestone doc
crux project add <name> [--type]   Register a new project (any type)
crux project link                  Link current repo to existing project by ID
crux export [--csv]                Export project data to CSV (for manual Sheets import)
crux sync --target sheets          Push snapshot to Google Sheets
crux ask "<question>"              Route to local LLM with DB context
crux ui [--port 8765] [--no-open]  Start browser UI server
```

## Three-Tier Routing

```
Tier 1 — crux CLI (terminal)            free, instant, no AI
  → status, reports, task updates, sync, session tracking, graph

Tier 2 — local LLM via crux ask         free, local, good enough for simple reasoning
  → "what should I work on today?"
  → "summarise my week"
  → "is this project worth continuing?"
  → "what's blocking the critical path?"

Tier 3 — Claude via MCP (this panel)    paid, cloud, heavy intelligence only
  → strategy under competing constraints
  → architecture decisions
  → multi-project trade-offs with ambiguous context
```

The Claude skill actively routes Tier 1 and Tier 2 queries away from Claude.

## Local LLM Integration (`crux ask`)

Config in `~/.crux/config.json`:

```json
{
  "llm": {
    "endpoint": "http://localhost:8080/v1/chat/completions",
    "model": "llama3.2",
    "max_tokens": 512
  }
}
```

Compatible with llama-server, Ollama, LM Studio — anything OpenAI-compatible.
Implementation: single `fetch()` call in `lib/ask.ts`. Zero npm deps.

## Browser UI (`crux ui`)

Local HTTP server via `node:http`. Binds to `127.0.0.1` only (no external exposure).

| Route | Content |
|---|---|
| `GET /` | Meta Kanban — all projects |
| `GET /project/:id` | Project detail: tasks, CPM, burndown |
| `GET /roi` | ROI dashboard + spread indicator |
| `GET /graph/:id` | Dependency DAG (SVG) |
| `GET /db` | Raw table explorer (power user) |
| `GET /api/*` | JSON endpoints (same data, scriptable) |

v1: read-only. All mutations via CLI or Claude MCP.
v2 (future): interactive — drag cards, click to update, inline ROI entry.

Default port: `8765`. Configurable via `--port`. `--no-open` suppresses browser launch.

## GitHub Integration

Uses the `gh` CLI (shells out). No Octokit, no GitHub API client. Features:

- Issue creation/update/close per task
- Label `pm-test` on all test-created resources (teardown sweep)
- Project Kanban board sync (create card, move card)
- Milestone creation on phase completion
- All sync is opt-in per project (`gh_sync: true`)

GitHub is one optional integration, not a requirement. Non-repo projects (articles, research) do not need it.

## Google Sheets Sync

Raw REST API via native `fetch()`. No `googleapis` package.
One-time OAuth2 setup: user drops `~/.crux/google-credentials.json`.

`crux sync --target sheets` pushes a structured snapshot:

- One sheet per project
- Tabs: overview, tasks, sessions, ROI
- Read-only from Sheets — DB remains source of truth

`crux export --csv` is the zero-friction alternative.

## Skill Auto-Install (`crux_init`)

During `crux_init`, after seeding the DB, the server checks for `.claude/skills/crux/SKILL.md`. If absent, it prompts:

```
crux works better with a Claude routing skill.
Add it to .claude/skills/? [y/N]
```

If yes, writes the skill from the embedded bundle. The skill embedded in the bundle is canonical — stays in sync with tool capabilities automatically.

## SKILL.md Routing Logic

```
ROUTING RULES — apply before every response:

TIER 1 (run directly, Bash tool, no reasoning):
→ status, reports, task updates, sync, session tracking, graph, ROI records
→ Bash: `crux <command>`

TIER 2 (local LLM sufficient, Bash tool):
→ "what next", "summarise", "is X worth it", "what's blocking"
→ Bash: `crux ask "<question>"`
→ relay the response verbatim

TIER 3 (full Claude intelligence required):
→ strategy across projects, architecture decisions, ambiguous priorities under constraints
→ run `crux overview` and `crux cpm` first to load current state, then reason
```

## Automated Pipeline Integration

Pre-commit hook example:

```
pre-commit
├── build + tests
├── crux test-run build pass|fail
├── crux test-run test-c pass|fail --coverage <pct> --task-slug <slug>
├── [on pass + coverage target met]
│   ├── crux task done <slug>         ← auto-close (no human needed)
│   ├── crux cpm                       ← recompute critical path
│   ├── crux report tasks              ← regenerate tasks.md
│   ├── crux sync --apply              ← push to GH Issues
│   ├── crux milestone check           ← detect phase completion
│   └── crux report status             ← audit trail doc
└── [on fail]
    └── crux test-run <phase> fail     ← log only, no closes
```

## ROI & Spread Model

**Inputs:**

- **Time:** `crux session start/end` — automatic minute tracking
- **Revenue:** `crux roi add <amount> --kind revenue|cost|expected`
- **Baseline rate:** set once in `~/.crux/config.json` as `"hourly_rate": 150`

**Signals:**

- `ROI score = total_revenue / hours_invested`
- If `ROI < baseline_rate` AND `hours_invested > threshold` → flag
- If `last_activity > 14 days` AND `status = active` → flag as stalled
- If `active_project_count > 2` → spread warning

**Output (`crux spread`):**

```
PROJECT                TYPE       STATUS    HOURS   ROI        LAST ACTIVE
─────────────────────────────────────────────────────────────────────────
hedge-fund             code       active    42h     —          today
article-on-cpm         article    active    8h      —          3 days ago
my-saas-project        code       stalled   95h     $4/hr ⚠   14 days ago

⚠  my-saas-project: 14 days inactive, ROI $4/hr vs $150/hr opportunity cost
⚠  SPREAD: 3 simultaneously active projects — focus peak is typically 2
```

## Integration Test Strategy

Dedicated test repo: `sholtomaud/crux-test` (persistent, state reset each test run).

All test-created GH resources labelled `pm-test`. Teardown:

```bash
gh issue list --repo crux-test --label pm-test --json number \
  | jq '.[].number' | xargs -I{} gh issue delete {} --repo crux-test --yes
```

**Unit tests** (`test/unit/`) — no network, no GH:

- CPM algorithm correctness (forward/backward pass, cycle detection, topo sort)
- DB schema and queries
- `tasks.md` parser
- Report generation output

**Integration tests** (`test/integration/`) — real GH, isolated repo:

- `sync.test.ts` — issue create/update/close
- `milestone.test.ts` — phase completion detection + milestone doc
- `coverage.test.ts` — auto-close on coverage target met
